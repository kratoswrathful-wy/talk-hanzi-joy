## 各項詳細說明

### 1. 多檔案合併開啟

**目標**：讓譯者能同時開啟多個檔案，在單一編輯器介面中處理，匯出時再依原始檔分組輸出。

**主要挑戰**：
- 句段 ID 命名空間衝突（不同檔案可能有重複的 rowIdx）
- 進度條與篩選邏輯須支援跨檔案聚合
- 架構改動涉及 `currentSegmentsList`、`openEditor`、IndexedDB / Supabase 儲存層

**建議**：架構改動最大，建議最後實作。

---

### 2. Trados 狀態區隔（sdlxliff）

**目標**：區分 sdlxliff 原生鎖定（`<sdl:seg conf="...">` 或 `locked="true"`）與譯者手動鎖定，顯示不同圖示，並在匯出時保留原始鎖定標記。

**主要挑戰**：
- `xliff-import.js` 需讀取 sdlxliff `<sdl:seg>` 屬性並設 `isLockedSystem`
- UI 區分 `isLockedUser`（🔒 藍色）與 `isLockedSystem`（🔒 橘色/灰色）
- 匯出時系統鎖定段落保留原始 XML 屬性不被覆蓋

---

### 3. AI 輔助機翻

**目標**：整合 AI API，提供批次機翻功能，支援文風學習與模組化提示詞。

**主要挑戰**：
- API 架構選擇（OpenAI / Anthropic / 自訂）
- 提示詞模組化設計（術語表、文風範例、格式指令）
- 安全性：API key 不可暴露於前端，需透過 Supabase Edge Function 中繼
- 批次處理（每批 10–15 行）與進度顯示

---

### 4. 作業階段區隔

**目標**：支援翻譯→校對→審閱等多階段作業，各階段有獨立完成狀態與指派記錄。

**主要挑戰**：
- 需先定義「階段」的語意（以專案為單位 vs. 以檔案為單位）
- DB schema 需新增 `stage` 相關欄位
- UI 需讓 PM 設定階段順序與人員指派

---

### 5. 整合 TMS

**目標**：CAT 工具與外部 TMS 案件系統連結，支援在 CAT 內寫入 TMS 備註，完成後自動同步狀態。

**主要挑戰**：
- 需先建立 CAT 案件與 TMS 案件的資料關聯
- postMessage 或 API 通訊協定設計
- 影響範圍高，需謹慎設計介面邊界

---

### 6. QA 功能

**目標**：提供靜態 QA 檢查，讓譯者在提交前發現常見錯誤。

**已完成（基礎）**：
- ✓ Tag 完整性檢查（`sourceTags` vs `targetTags` id 集合比對）
- ✓ TB 術語未套用檢查（原文含術語但譯文未含對應翻譯）
- ✓ 分頁已從 Comments 更名為 QA
- ✓ 結果表格支援點擊跳轉、忽略標記、隱藏已忽略

**待補充（進階規則）**：
- 數字一致性（原文數字是否出現於譯文）
- 標點符號一致性（句末標點、引號風格）
- 未翻譯句段（原文與譯文相同且非專有名詞）
- 大小寫一致性

---

### 7. 派案方準則 ✓ 完成

**原始估計**：小。**實際範圍**：大。

**已落地為「筆記與共用資訊重設計」**，包含：
- Quill 富文字私人筆記（個人可見，每個 CAT 檔案獨立）
- 待辦事項（`item_type: 'todo'`，可標記完成）
- 共用準則 CRUD + 版本歷史（PM 可建立，全員唯讀瀏覽）
- 討論串（以句段為單位的留言）
- Supabase 整合（`cat_private_notes`、`cat_shared_guidelines` 表格）

---

### 8. 粗體/斜體格式功能（新增）

**目標**：XLIFF/mqxliff/Excel 匯入時，若原文含粗體/斜體，直接以視覺渲染（`<b>`/`<i>`）顯示，而非以 tag pill 表示；並提供 Ctrl+B/Ctrl+I 或工具列按鈕讓譯者在譯文中套用格式。

**方案**：方案二（視覺渲染 + Ctrl+B/I 插入新格式），優先處理 XLIFF/mqxliff；其他格式（字體顏色、背景套色等）仍以 tag pill 處理。

**主要挑戰**：
- 各格式的粗體/斜體標記不同（XLIFF `<g ctype="bold">`、mqxliff `{b}`/`{/b}` 或 `<ph type="bold"/>`、Excel `runs[].font.bold`）
- 匯出時須將 `<b>`/`<i>` DOM 轉回原始格式
- contenteditable + Selection API 的游標處理

---

## 開發優先順序建議

依工作量小、影響範圍低、獨立性高的原則排序：

| 優先 | 功能 | 理由 |
|------|------|------|
| 高 | 2. Trados 狀態區隔 | 獨立模組，風險低，Trados 使用者高需求 |
| 高 | 6. QA 進階規則 | 基礎已完成，補充靜態規則即可 |
| 中 | 8. 粗體/斜體格式 | 改善日常體驗，範圍可控 |
| 中 | 4. 作業階段區隔 | 需先定義語意，架構影響中等 |
| 中 | 3. AI 輔助機翻 | 需先決定 API 架構 |
| 低 | 5. 整合 TMS | 需先建立資料關聯，影響範圍高 |
| 最後 | 1. 多檔案合併開啟 | 架構改動最大，建議最後做 |
